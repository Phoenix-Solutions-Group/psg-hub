## THE BLOCKER — MCC unsupported (Phase 10 link)

After OAuth consent, the callback (`src/app/api/ads/google/callback/route.ts:120-124`) **hard-errors when `listAccessibleCustomers` returns >1 customer**: "single-customer link only in this release." PSG's Google account is an **MCC (manager)** → sees many customers → **no real PSG client can link as-is.** This is a genuine product gap, not config.

**Operator decision (Nick, this session): "add MCC support for sure" + "the system should allow the user to choose which accounts are visible, at authentication."** → Build an account-selection step in the OAuth link.

### Design sketch (NOT yet built — see open question first)
- Single account per shop (multiple accounts/shop would collide on the `analytics_snapshots (shop_id,source,date,period)` key — a larger ingest change).
- Migration: transient `pending_*` columns on `google_ads_oauth_states` (encrypted token, key_version, scope, login_customer_id, customers[]).
- `callback`: on >1, encrypt token, stash in state row (don't consume), render account-picker page.
- New `POST /api/ads/google/select`: re-bind session (user==state.userId), consume state, validate pick ∈ offered list, persist chosen account with `login_customer_id=<MCC>`.
- `client.ts:204` already passes `login_customer_id` → metrics sync works for MCC children once stored.

### ⚠️ OPEN QUESTION that gates the design (advisor catch + RESEARCH.md:155)
`listAccessibleCustomers` returns customers **directly accessible** to the user. For an MCC login that is often **the manager account(s), NOT the child accounts**. If so, Wallace `6048611995` may not appear in that list and the picker must enumerate children via a `customer_client` GAQL query under `login-customer-id=<MCC>` — a different data source. **VERIFY (context7/Google Ads API docs) before building the picker.** RESEARCH.md line 155 already lists this as open.
