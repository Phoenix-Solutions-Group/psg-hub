<objective>
## Goal
Wire Supabase auth into the Next.js app for the project `gylkkzmcmbdftxieyabw`. Magic-link login flow at `/login` works end-to-end. A `user_profile` table exists with `role` (`psg_admin` | `client`) + `client_id` columns. A Supabase Auth custom-access-token hook injects `role` and `client_id` as JWT custom claims. An authed layout group protects subsequent routes.

## Purpose
RLS in Phase 2+ depends on JWT claims. Without those claims, no multi-tenant query is safe. Locking the auth + claims pattern here means Phase 2's RLS policies have something correct to read against.

## Output
- Supabase clients (browser, server, middleware) using `@supabase/ssr`
- `supabase/migrations/01_user_profile.sql` — table + RLS + trigger
- `supabase/functions/custom-access-token-hook` — Edge Function that reads `user_profile` and injects `role` + `client_id` claims (registered as Supabase Auth hook)
- Magic-link login flow: `/login` form → email sent → callback exchanges code → session set
- `/auth/signout` route handler for sign-out
- `app/(authed)/layout.tsx` route-group layout that gates protected routes via server-side session check
- `.env.example` updated; `.env.local` populated locally (not committed); Vercel env vars set for the three Supabase secrets

<task type="auto">
  <name>Task 2: Supabase migration + custom-access-token hook</name>
  <files>supabase/migrations/01_user_profile.sql, supabase/functions/custom-access-token-hook/index.ts</files>
  <action>
    1. Initialize Supabase CLI in repo if not already: `pnpm dlx supabase init`. Link to project: `pnpm dlx supabase link --project-ref gylkkzmcmbdftxieyabw`.

    2. Create migration `supabase/migrations/01_user_profile.sql`:
       ```sql
       create table public.user_profile (
         id uuid primary key references auth.users (id) on delete cascade,
         role text not null default 'client' check (role in ('psg_admin', 'client')),
         client_id uuid references public.client (id),
         display_name text,
         created_at timestamptz not null default now()
       );

       alter table public.user_profile enable row level security;

       create policy "user reads own profile" on public.user_profile
         for select using (auth.uid() = id);

       create policy "psg_admin reads all profiles" on public.user_profile
         for select using (
           exists (select 1 from public.user_profile p
                   where p.id = auth.uid() and p.role = 'psg_admin')
         );

       -- trigger to auto-create user_profile on signup
       create or replace function public.handle_new_user()
       returns trigger language plpgsql security definer as $$
       begin
         insert into public.user_profile (id) values (new.id);
         return new;
       end; $$;

       drop trigger if exists on_auth_user_created on auth.users;
       create trigger on_auth_user_created
         after insert on auth.users
         for each row execute function public.handle_new_user();
       ```
       NOTE: `client` table doesn't exist yet (Phase 2). For now, make `client_id` nullable with no FK; or create a stub `client` table in this migration so the FK works. PREFER the stub.

    3. Create Edge Function `supabase/functions/custom-access-token-hook/index.ts`:
       - Receives auth event { user_id, claims }
       - Looks up user_profile row for user_id (using service-role inside the function context)
       - Returns claims merged with { role, client_id }

    4. Deploy:
       - `pnpm dlx supabase db push` for the migration
       - `pnpm dlx supabase functions deploy custom-access-token-hook --no-verify-jwt`

    5. In Supabase dashboard → Auth → Hooks → Custom Access Token, register the deployed function.

    Avoid:

    - Leaving the hook unregistered (the migration alone doesn't activate it)
    - Forgetting the stub `client` table — Phase 2 will extend it
