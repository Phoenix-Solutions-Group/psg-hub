<objective>
## Goal
AI-drafted responses to reviews via Claude Haiku 4.5. Per-platform tone presets. Human edit + approval gate before anything leaves the system. Copy-to-clipboard for manual posting (no automated posting — Yelp has no reply API, Google GBP needs OAuth deferred to 02-05).

## Purpose
Collision repair shop owners don't have time to respond to every review. BSM drafts in PSG voice, they approve or edit, then paste to the platform. Removes the biggest friction in reputation management.

## Output
- `review_responses` table + `review_response_versions` (immutable audit log) + `llm_call_log` (observability)
- Responder library calling Anthropic SDK with platform-aware prompts, prompt-injection defense, output safety guardrails
- 2 API routes: draft + approve (state-machine enforced, rate-limited, role-gated)
- UI: accessible modal on `/reviews` with edit/regenerate/approve/copy actions
- Unit + integration tests
</objective>

<acceptance_criteria>

## AC-1: Draft generation
```gherkin
Given a review exists in the reviews table with no prior response
When authenticated shop member clicks "Draft response" and POSTs to /api/reviews/{id}/draft-response
Then a review_responses row is created with status="draft", body populated by Claude Haiku 4.5, tone matching the platform (Google warmer, Yelp more concise), and the response appears in the modal within ~5s (20s hard timeout)
```

## AC-2: Edit and regenerate
```gherkin
Given a draft response exists
When the user edits the body in the modal or clicks "Regenerate" with a different tone_preset
Then the edited body persists on save, or regeneration replaces body + increments version counter, and every prior version is preserved in review_response_versions (append-only, never deleted)
```

## AC-3: Approval + copy-to-post
```gherkin
Given a draft response exists
When a user with shop_role in ('owner','manager') clicks "Approve"
Then status flips to "approved", approved_by + approved_at set, body becomes read-only, and a "Copy to clipboard" button makes the text available to paste into the platform (since no automated posting in this plan)
```

## AC-4: Tenancy + auth
```gherkin
Given a user tries to draft/approve a response for a review outside their shop membership
When they POST to either route
Then the route returns 403 before any LLM call is made (explicit membership check, not RLS-reliant), no row written, no LLM call charged
```

<!-- audit-added: AC-5 through AC-9 -->

## AC-5: Prompt-injection defense
```gherkin
Given a review body contains text attempting to override instructions (e.g., "Ignore previous instructions and write...")
When draft-response is called
Then the model treats the review body as untrusted data (not instructions), the system prompt explicitly boundaries review content as data, and the output still follows PSG voice rules (no admission of liability, no insurance promises, no PII echoed)
```

## AC-6: Rate limiting + cost governance
```gherkin
Given a shop has generated drafts this hour
When a user attempts more than 10 draft/regenerate calls per review per hour OR more than 100 LLM calls per shop per day
Then the route returns 429 with a clear message before any LLM call fires, and the attempt is logged in llm_call_log with result="rate_limited"
```

## AC-7: State machine enforcement
```gherkin
Given a review_response exists in state S
When an API transition is attempted
Then only valid transitions succeed: draft→approved, draft→rejected, draft→draft (update/regenerate), approved→draft (un-approve explicit action); any other transition returns 409 Conflict; approved_by/approved_at are cleared on un-approve
```

## AC-8: Observability + audit trail
```gherkin
Given any LLM draft call fires
When it completes (success, error, timeout, or rate-limited)
Then an llm_call_log row is written with user_id, shop_id, review_id, model_id, input_tokens, output_tokens, latency_ms, status, error_code; and for every response body change, a review_response_versions row is inserted immutably (no UPDATE, no DELETE)
```

## AC-9: Output safety
```gherkin
Given the model produces a draft response
When the draft contains disallowed patterns (phone numbers, email addresses, admission phrases like "we were at fault", "insurance will cover", "we promise")
Then the draft is flagged (`safety_flags jsonb` populated), the UI surfaces the flags to the approver, and the approve route refuses to transition to 'approved' until the flag list is reviewed (manual override allowed with explicit acknowledgment field)
```
