## Grounding (plan-time, decisive)
1. **`analytics_sync_runs` table does NOT exist** — typed in 09-01's types.ts but never migrated. This plan creates it (migrations-as-code per PROTOCOL; LOCAL-applied; joins the gate-batch prod apply).
2. **PERIOD RECONCILE:** research suggested `period='monthly'` rows, but the 09-02 page reads `period='daily'` over a trailing-30-day window. Ingest writes **`period='daily'`, `date=today UTC`** — each sync day = one time-series point (metric drift over days), same-day re-runs idempotently overwrite. Monthly rollups can be derived later for the Phase-12 report.
3. **Cost guard:** `domain_organic` bills 10 units/row. The research's `display_limit=1000` = 10k units per shop per sync — unaffordable daily. Cap at **`display_limit=100`** (top-100 by traffic sample, ~1k units/shop) and run the cron **daily** (06:00 UTC), not the generic 6h canon — SEMrush metrics are monthly estimates; intra-day refresh buys nothing. Per-sync cost ≈ 4 shops × (~50 + ~1000) ≈ 4.2k units/day.

## AC-1: Contract-correct SEMrush client
```gherkin
Given the live-verified API contract (research/semrush-api.md)
When the client fetches a domain's metrics
Then it calls domain_rank (Or/Ot/Oc) + backlinks_overview (total + score — NOT ascore)
  + domain_organic capped at display_limit=100 bucketed client-side into Po top3/top10/top20/top100
And every response is parsed BY THE RETURNED HEADER (never positional by request order)
And a leading ERROR token (even with HTTP 200) raises a typed error, never parses as data
And all calls run through withRetry + a shared CircuitBreaker (resilience constraint)
```

## AC-2: Idempotent per-shop sync with audit ledger
```gherkin
Given shops where url IS NOT NULL (4 of 7 on prod)
When syncSemrushSnapshots runs
Then each such shop gets ONE analytics_snapshots row (source='semrush', period='daily',
  date=today UTC, metrics=SemrushMetrics) via upsertSnapshots — re-running the same day nets zero new rows
And shops without a url are SKIPPED (counted, no API call, no row — their no-data state stands)
And one analytics_sync_runs ledger row records the run (running → success with rows_written,
  or error with the message); a single shop's failure doesn't abort the other shops
```

## AC-3: Secret-gated cron route
```gherkin
Given POST /api/cron/semrush-sync
When called without Authorization: Bearer ${CRON_SECRET} (or with a wrong secret)
Then 401, no sync, no API spend
When called with the correct secret
Then the sync runs and the response reports { synced, skipped, failed } counts
And vercel.json declares the daily cron (06:00 UTC) for this path
```
