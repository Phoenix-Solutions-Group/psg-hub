<objective>
## Goal
Settle the three Phase-7 carry-in surface defects so the live customer shell is
internally consistent before the PII/RLS and quality-gate plans review final state:
(1) the Ads page honors the 07-03 active-shop cookie instead of its own owner-first
default; (2) the dashboard nav + shop switcher are reachable on mobile, not just
`lg:` and up; (3) the dashboard home stops advertising a phantom Agent Runs metric
and shows real active-shop counts.

## Purpose
Phase 8 must make the surface safe and honest for a real shop to log in (PROJECT.md
"Active"). These three are the visible incoherences a pilot shop (Wallace / Tedesco /
Tracy's) would hit first: a switched shop that silently reverts on the Ads tab, a
mobile user with no nav and no way to switch shops, and a home screen showing a fake
metric for a feature (agents, v1.6) that does not exist yet. Clearing them is
code-only and unblocks 08-02 (PII/RLS) from auditing a moving surface.

## Output
- `src/app/dashboard/ads/page.tsx` — no-param default routed through `getActiveShopContext`.
- `src/components/dashboard/mobile-nav.tsx` — new client mobile nav (NAV + `<ShopSwitcher>`).

- `.paul/phases/08-launch-hardening/08-01-SUMMARY.md`.

<acceptance_criteria>

## AC-1: Ads page honors the active shop
```gherkin
Given a multi-shop user who switched their active shop to Shop B via the switcher
And they navigate to /dashboard/ads with NO shop_id query param
When the page resolves the default shop
Then it redirects to /dashboard/ads?shop_id=<Shop B id> (the active-shop cookie value)
And NOT to their owner-first shop
```

## AC-2: Explicit shop_id still wins and stays membership-gated
```gherkin
Given any user requests /dashboard/ads?shop_id=<X>
When X is a shop the user is a member of
Then the page proceeds with shop X (explicit param wins over the cookie)
And when X is NOT a shop the user belongs to
Then the page redirects to /dashboard (the existing membership re-validation is unchanged)
```

## AC-3: Mobile nav exposes navigation and the shop switcher below lg
```gherkin
Given a signed-in dashboard user on a viewport narrower than lg (sidebar hidden)
When the dashboard shell renders
Then a mobile nav control is present in the header (hidden at lg and up)
And opening it reveals the same NAV links as the desktop sidebar
And the <ShopSwitcher> (when the user has >=1 shop) so a shop can be switched on mobile
```

## AC-4: Dashboard home shows no phantom Agent Runs metric
```gherkin
Given the dashboard home page
When it renders the stat cards
Then there is no "Agent Runs" card (agents are deferred to v1.6; no agent_runs table)
And the remaining cards reflect counts scoped to the active shop (0 when no active shop)
And no query targets a phantom table (no .from("agent_runs"))
```
