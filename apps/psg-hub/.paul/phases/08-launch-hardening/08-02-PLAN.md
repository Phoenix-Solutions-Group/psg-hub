<objective>
## Goal
Close the blanket `Allow all` RLS hole on the 12 shared multi-tenant `public` tables on prod `gylkkzmcmbdftxieyabw` — for BOTH the `anon` AND the `authenticated` role — so a logged-in shop can no longer read or write another tenant's rows. Replace each blanket policy with either a scoped `to authenticated` policy or a confirmed default-deny (RLS-on, no policy = service-role-only), table by table, under the 06-01 PROTOCOL.

## Purpose
This is the M2 PII gate and the literal Phase-8 entry condition ("safe for a real shop to log in with live PII"). The ROADMAP framed this as "26 anon-open policies," but grounding (2026-06-03, read-only `pg_policies`) showed the holes are `Allow all` for `anon` AND `authenticated` — the `authenticated` half is a cross-tenant breach: today any logged-in user can read/write every shop's rows in `profiles`, `shops`, `clients`, `reviews`, and 8 more. Pilot onboarding (Wallace/Tedesco/Tracy's) cannot proceed onto live data until this is closed. Resolves D3 (deferred from 06-01, FIXED-to-Phase-8).

## Output
- One migrations-as-code file dropping the blanket policies and installing scoped replacements / confirmed default-deny per the Task-1 verdict.
- Post-migration advisor diff (re-captured baseline) proving no new ERROR/WARN and a dropped `rls_policy_always_true` count.
- Live verification: cross-tenant denial proven; psg-hub member flows unbroken; anon denied.
</objective>

## Grounding captured at plan time (2026-06-03, read-only MCP)

**The 12 tables carrying blanket `Allow all` (qual=true, with_check=true):**
`activity_log`, `campaigns`, `clients`, `configs`, `discovery_briefs`, `elements`, `pages`, `profiles`, `research_artifacts`, `reviews`, `shops`, `skills`.
Each has the pair `Allow all for anon [ALL/{anon}]` + `Allow all for authenticated [ALL/{authenticated}]`.
(`reports` has a `service_role` qual=true policy — benign, service-role bypasses RLS regardless; NOT in scope. `review_items`/`review_responses` are already properly scoped by 06-04 — NOT in scope.)

**psg-hub source touches ONLY 2 of the 12** (grep `.from("<table>")` across `src/`):
- `shops` — read by user-session pages (`dashboard/settings/page.tsx:22`, `dashboard/ads/page.tsx:58`, `api/reviews/[id]/draft-response/route.ts:99`) AND service-role (`lib/tier/gate.ts:55`, `lib/google-ads/tier.ts:11`, `api/ads/google/campaigns/route.ts:124`, `api/webhooks/stripe/route.ts:51`, `api/onboarding/route.ts` service writes).
- `clients` — `api/onboarding/route.ts` via the **service client only** (no user-session read).
- The other 10 tables (`profiles`, `reviews`, `campaigns`, `configs`, `activity_log`, `discovery_briefs`, `elements`, `pages`, `research_artifacts`, `skills`): **zero psg-hub references.** psg-hub reads identity/role via `app_user_roles` + `shop_users` (06-02/03), not `profiles`.

**Per-table starting state (drives the verdict):**
- `shops`: blanket(anon+auth) PLUS scoped survivors `shops_select` (`id in (select user_shop_ids())`) and `shops_update` (same, with_check same), both currently `to {public}`. → Dropping the blanket leaves working scoped reads; anon (auth.uid() null → empty shop set) is denied. **Pre-decided: drop blanket, retighten survivors `{public}`→`{authenticated}`.**
- `clients`: blanket ONLY. psg-hub uses service-role only. **Pre-decided: drop both blanket → default-deny (RLS-on, no policy).**
- `profiles`: blanket ONLY; highest-PII; auth-critical (`handle_new_user` writes via SECURITY DEFINER trigger = RLS-bypassing; not affected by dropping a policy). **Pre-decided: drop blanket + add self-row `to authenticated` SELECT/UPDATE (`id = (select auth.uid())`); optional superadmin-read via `private.current_user_role()`.**
- The 9 remaining sibling/agentic tables (`campaigns`, `configs`, `activity_log`, `discovery_briefs`, `elements`, `pages`, `research_artifacts`, `skills`, `reviews`): psg-hub never reads them. External anon/sibling-app dependence is the only open question (local_reach archived Phase 5; old `data` portal project deleted Phase 3). **→ Task-1 checkpoint resolves per table.**

**Permissive-OR reminder:** Postgres RLS combines policies with OR. While a blanket `Allow all` exists, every scoped policy on that table is dead weight — the blanket alone grants access. The breach closes only when the blanket is dropped.

**Advisor baseline note:** PROTOCOL §4 baseline is a dated snapshot; `rls_policy_always_true` ≈ 26 findings (the anon+auth pairs across the 12 tables + benign `reports`). Task 3 RE-CAPTURES `get_advisors(security)` at apply time and diffs against that fresh capture, per PROTOCOL §4 — not against the stale table.
