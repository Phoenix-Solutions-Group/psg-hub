<objective>
## Goal
Build the financial-record-only invoice + payment mirror: two new tables (Stripe IDs + amounts/dates/status, NO PII columns) with the membership-SELECT / service-write RLS pattern, plus the `invoice.*` and `payment_intent.*` webhook handlers that populate them, gated by the 15-01 idempotency spine and using the Basil field relocations. Build-local; migration authored-not-applied.

## Purpose
Phase 17 (Stripe-native invoices + payment links) surfaces this data; Phase 15 lays the model and the ingest path so 17 is a UI/surface phase, not a from-scratch build. Financial-record-only is the operator decision (2026-06-18): billing name/email/address stay in Stripe, fetched on demand, so this plan adds no encrypted PII columns. The handlers must use the post-Basil invoice→subscription and invoice→payment field paths that the pre-Basil code never had.

## Output
One authored-not-applied migration (`invoices`, `payments` tables + RLS), new `invoice.{created,finalized,paid,payment_failed}` + `payment_intent.{succeeded,payment_failed}` handlers in the webhook switch, `Invoice`/`Payment` TS types, and extended webhook tests.
</objective>
