<objective>
## Goal
Harden the inherited Stripe webhook into a correct, idempotent, Basil-safe spine: add an inbound event-idempotency table, gate every event on it, fix the S3 `.insert()` defect, fix the latent Basil `current_period_end` read, wrap outbound Stripe calls in the shared resilience util, and reconcile the vestigial `shops.subscription_tier` + the `multi_location` tier mismatch. Build-local only — migrations authored, not applied.

## Purpose
Phases 16 (subscription self-serve) and 17 (Stripe-native invoices) both build on this webhook. Today it carries three confirmed defects that silently corrupt the tier `src/lib/tier/gate.ts` gates on: the S3 redelivery duplicate-key swallow (`route.ts:56`), the Basil sub-level `current_period_end` read that returns `undefined` under the pinned `2026-05-27.dahlia` API (`route.ts:71`), and no event-idempotency gate (Stripe is at-least-once; every redelivery reprocesses). Fixing the spine before extending it stops the Phase-10 blind-build pattern from repeating.

## Output
Two authored-not-applied migrations (`stripe_webhook_events` idempotency table; subscription-tier reconcile), a hardened `webhooks/stripe/route.ts`, a module-level `StripeCircuitBreaker` + retry-wrapped retrieve helper in `stripe.ts`, and a webhook test suite proving idempotency + upsert + the Basil fix.
</objective>

<acceptance_criteria>

## AC-1: Event idempotency gate
```gherkin
Given the webhook has already processed a Stripe event with a given event_id
When Stripe redelivers the same event_id (at-least-once delivery)
Then the handler records the event once via INSERT ... ON CONFLICT (event_id) DO NOTHING,
  detects the empty RETURNING result, skips all side effects, and returns 200
  with zero net database changes
```

## AC-2: S3 subscription upsert (no duplicate-key swallow)
```gherkin
Given a shop already has a subscriptions row
When checkout.session.completed fires again (re-subscribe or redelivery past the event gate)
Then the subscription row is upserted in place (one shop ↔ one subscription; shop_id is UNIQUE),
  the tier updates to the new value rather than raising a silently-swallowed duplicate-key,
  and any write error is checked and surfaced (no bare .insert() without error handling)
```

## AC-3: Basil current_period_end read
```gherkin
Given a customer.subscription.updated event under apiVersion 2026-05-27.dahlia
When the handler computes the access-expiry period end
Then it reads from subscription.items.data[0].current_period_end (post-Basil location),
  not the pre-Basil subscription.current_period_end (which is undefined under this version),
  and persists a non-null period end for an active subscription
```

## AC-4: Resilience + tier reconcile
```gherkin
Given the webhook makes an outbound Stripe API call (e.g. subscriptions.retrieve for a stale-prone payload)
When the call is issued
Then it runs through withRetry + a shared module-level StripeCircuitBreaker (mirroring the SendGrid/Twilio/Google adapters)

Given the subscriptions.tier TS enum has 3 values and shops.subscription_tier is read by no source code
When the reconcile migration is authored
Then it drops the vestigial shops.subscription_tier column and tightens both tier CHECK constraints
  to the 3-value set (essentials, growth, performance), removing the unused multi_location mismatch
```
