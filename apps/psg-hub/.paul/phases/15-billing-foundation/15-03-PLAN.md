<objective>
## Goal
Stand up the billing PII-at-rest infrastructure — a private `sensitive` schema with access logging and an on-demand 7-year retention sweep, a generic column-encryption wrapper over the existing AES-256-GCM util, and a binding retention/taxonomy spec — WITHOUT storing any populated PII column. Infrastructure only; build-local; migration authored-not-applied.

## Purpose
The roadmap picks up the v0.3-deferred PII-at-rest retention here. Per the operator decision (2026-06-18), invoices/payments are financial-record-only (Stripe IDs + amounts/dates/status); billing name/email/address stay in Stripe and are fetched on demand. So Phase 15 ships the encryption + access-log + retention machinery that 15-02 and Phase 17 will draw on, but creates no populated encrypted columns now (nothing stores customer PII until a real subscribe, which is Phase-18-gated — populated columns now would be speculative). This mirrors the proven internal precedent rather than inventing a scheme.

## Output
One authored-not-applied migration (private `sensitive` schema + `pii_access_log` + `log_pii_access` SECURITY DEFINER + on-demand retention sweep function), a generic `encryptField`/`decryptField` wrapper, a roundtrip test, and `PII-RETENTION.md` as the binding taxonomy + retention spec for downstream billing plans.
</objective>
