<objective>
## Goal
Build the GBP reply publish-to-Google vertical (legacy v4 `updateReply`/`deleteReply`) build-local: a publish-state migration on `review_responses`, a raw-HTTP reply adapter, and a publish orchestrator + an UNSCHEDULED cron — all wired so that NO user-reachable, deploy-live publish trigger ships. Surface the load-bearing consent/authorization model as a checkpoint:decision that gates live activation. Build-local, ZERO prod, ZERO live publish.

## Purpose
14-01 fills `review_items` and the draft+approve pipeline already exists, but nothing publishes an approved reply to Google. Per 14-RESEARCH the publish step is net-new AND carries a load-bearing LEGAL/PRODUCT gap: Google's policy bars "automate OR trigger" replies without the end-client's prior specific express consent (a human Approve click is a trigger; the click counts only if it is the shop owner/manager's own per-reply recorded consent), and an on-behalf-of agency must capture a per-shop end-client authorization first. The current `approve-response` gate allows a PSG operator. This plan therefore builds only the consent-NEUTRAL plumbing and refuses to ship any path that could publish an operator-approved reply, deferring activation + the consent records + who-clicks tightening behind a recorded decision (operator chose "14-02 now, build-local").

## Output
- One additive, idempotent migration: `review_responses` gains `publish_status` + `publish_error` + `publish_attempts` + `published_version` + `external_reply_updated_at` (the `status` draft/approved/rejected state machine and the approve-response route are UNTOUCHED).
- `gbp-reviews-reply.ts` — raw-HTTP v4 `updateReply` (+ `deleteReply` rollback) adapter, mirroring the `gbp-reviews.ts` seam.
- `gbp-reviews-reply-sync.ts` publish orchestrator + an UNSCHEDULED `/api/cron/gbp-reviews-reply` cron (NOT added to vercel.json). NO manual API route, NO UI button (both gated behind the consent decision).
</objective>

## AC-1: The reply adapter PUTs updateReply correctly and enforces the byte limit PSG-side
```gherkin
Given a linked gbp account with a non-null externalParentId and a review external_review_id
When publishReply(shopId, reviewName, comment, deps) is called with a valid comment
Then it issues PUT v4/{reviewName}/reply with body { comment } ONLY (no output-only fields), to the plain slash-join parent (no double-prefix),
  reads back the output-only reviewReplyState, and a comment whose Buffer.byteLength(comment,'utf8') exceeds 4096 is rejected PSG-side BEFORE any request (never string .length)
And the module also exports deleteReply(reviewName, deps) issuing DELETE v4/{reviewName}/reply with an empty body (the rollback path)
```

## AC-2: An unverified-location WRITE rejection degrades to publish_failed, never trips the breaker or flips the account
```gherkin
Given the publish orchestrator attempts a reply on a location that is not verified
When updateReply rejects (the v4 verified-location precondition; exact status pinned at gate-batch live-smoke)
Then the response row is set publish_status='publish_failed' with publish_error='location not verified' (mapped), publish_attempts is incremented,
  the account is NOT flipped to status='error' (a non-VoM location is a data-state, not an auth failure), and the batch continues
And NO pre-write hasVoiceOfMerchant short-circuit is required (the signal is not persisted post-13-01; the WRITE rejection is the gate — recorded deferral)
```

## AC-3: The publish orchestrator publishes approved, dirty rows idempotently with contained per-row failure
```gherkin
Given review_responses rows with status='approved'
When syncGbpReviewReplies(service) runs
Then it selects rows where status='approved' AND (published_version IS NULL OR published_version < version) AND publish_status <> 'publishing',
  resolves each row's review external_review_id + shop gbp account, PUTs the reply, and on success sets published_at, published_version=version,
  publish_status='published', external_reply_updated_at, and reflects the returned reviewReplyState (a PENDING state is NOT reported as 'published'),
  applies the update with optimistic version concurrency (mirrors approve-response), a re-run nets zero new publishes,
  a single row's failure is contained (publish_status='publish_failed' + publish_error + publish_attempts++; auth_failed flips that account via markAccountError)
```
