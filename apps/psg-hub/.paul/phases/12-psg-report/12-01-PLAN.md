<objective>
## Goal
Build the Phase-12 report DATA layer: a deterministic monthly-rollup engine over `analytics_snapshots` and a single `ReportData` payload assembler that turns the four live sources (SEMrush, Google Ads, GA4, GSC) into one typed object for a given shop and month. Pure functions, fully unit-tested, zero prod contact. No LLM, no PDF, no cron.

## Purpose
Phase 12 (the automated PSG monthly report) needs one correct structured input before any narrative or PDF work. The hard part is the daily-to-monthly rollup: every metric must be classified FLOW (sum), STOCK (latest-in-month), or DERIVED-RATIO (recompute from summed components), because the four sources store ONLY `period:'daily'` rows and a naive sum corrupts stocks and ratios. RESEARCH.md proves the trap (SEMrush `organic_traffic` is an estimated-monthly value re-snapshotted daily, so summing overcounts; `cpl`/`ctr`/`position`/`engagement_rate` are ratios that must be recomputed, never averaged). 12-02 (narrative + eval) and 12-03 (PDF) both consume `ReportData`, so its correctness gates the whole phase.

## Output
- `src/lib/analytics/rollup.ts`: `monthWindow`, the per-source metric-class registry, `rollupMonth`, `momDelta` (+ tests).
- `src/lib/report/types.ts`: `ReportData` and per-source report-block types.
- `src/lib/report/report-data.ts`: `assembleReportData(shopId, periodMonth, deps)` (+ tests).
</objective>

<acceptance_criteria>

## AC-1: The monthly rollup classifies and reduces every metric correctly
```gherkin
Given a set of daily analytics_snapshots rows for one shop, one source, within a calendar month
When rollupMonth(source, rows) runs
Then FLOW metrics are summed (Ads: spend/clicks/impressions/conversions/cost_micros; GA4: sessions/engaged_sessions/new_users/key_events; GSC: clicks/impressions)
And STOCK metrics take the latest-dated row's value, never a sum (all SEMrush metrics — organic_keywords/organic_traffic/organic_traffic_cost/backlinks/authority_score; and GA4 total_users/active_users are summed with a documented "approximate, overcounts uniques" caveat — true monthly uniques deferred to a monthly runReport)
And DERIVED-RATIO metrics are RECOMPUTED from summed components, never averaged: Ads cpl = sum(spend)/sum(conversions) (null when conversions=0), GSC ctr = sum(clicks)/sum(impressions), GSC position = sum(position_i * impressions_i)/sum(impressions_i) (impression-weighted), GA4 engagement_rate = sum(engaged_sessions)/sum(sessions)
And an empty row set yields null (no data), not a zero-filled object
```

## AC-2: ReportData is assembled per shop per month from the live snapshots
```gherkin
Given a shop with linked sources and stored daily snapshots spanning the report month and the prior month
When assembleReportData(shopId, periodMonth, deps) runs (deps.getSnapshots injected)
Then it fetches a 90-day trailing window per source ONCE, rolls the current month and the prior month per source via rollupMonth, and computes momDelta for each metric
And it returns a typed ReportData with: period (month), per-source blocks (current, prior, momDelta, trend series via toSeries), linkedSources, and sourcesWithPriorMonth
And momDelta returns a null-safe percentage (null when prior is null or zero, signed number otherwise)
```

## AC-3: Missing sources and cold-start months degrade gracefully
```gherkin
Given a shop linked to only some sources (e.g. GA4 + GSC, no Google Ads) and/or with no clean prior month
When assembleReportData runs
Then sources with zero rows are OMITTED from ReportData (not present-with-zeros), and linkedSources reflects only the sources that returned data
And when a source has a current month but no prior month, its block sets prior=null and momDelta=null and the report flags cold-start (sourcesWithPriorMonth excludes it), matching the canon "3-week progress" within-period framing default
```
