<objective>
## Goal
Build the Phase-12 narrative layer: a multi-LLM writer that turns `ReportData` (12-01) into a structured, brand-true monthly narrative, plus a deterministic eval gate that BLOCKS any narrative with a fabricated number, wrong MoM direction, cross-source mis-attribution, or em-dash/emoji before it can reach a PDF. Pure + unit-tested with mocked LLM calls; the live Gateway smoke is deferred to the 12-04 activation.

## Purpose
RESEARCH.md's load-bearing finding: a report narrative pinned to FIXED numbers fails on fabricated/drifted numbers and brand-rule violations, not bland prose. So the design is ONE writer (not best-of-N) plus a deterministic verifier that code enforces at 100%. The eval gate is the safety boundary for a client-facing deliverable: the cron (12-04) never renders or emails a narrative that has not passed. This plan builds the writer + gate + orchestrator; 12-03 renders the passed narrative to PDF, 12-04 schedules it.

## Output
- `report/schema.ts` (zod section schemas), `report/prompt.ts` (system/brand block + token-substitution), `report/narrative.ts` (writer), `report/evaluate.ts` (eval cascade), `report/generate.ts` (orchestrator) + tests.
- `ai` (v6) + `zod` added (the phase's first new deps).
</objective>

## AC-1: The writer produces a structured, grounded narrative via AI SDK v6 through the Gateway
```gherkin
Given a ReportData payload for a shop+month
When generateNarrative's writer runs
Then it calls AI SDK v6 generateText with Output.object({ schema }) routed through the Vercel AI Gateway by a bare `provider/model` string (writer default anthropic/claude-sonnet-4.6; anthropic/claude-opus-4.8 for headline + recommendation sections), with a same-family Claude fallback via providerOptions.gateway.models
And numbers are grounded by token substitution: the writer references metrics by placeholder (e.g. {{ga4_sessions_mom_pct}}) and code substitutes the real formatted values from ReportData AFTER generation, so the model never types a metric value
And brand rules are injected inline via zod field .describe() ("No em dashes. No emojis. Active voice.")
And the call is wrapped in withRetry + CircuitBreaker and logged via logLLMCall(purpose="monthly_report_narrative", reviewId=null, shopId=<shop>)
And the actual generate call is injected via deps so unit tests mock it; tsc compiles the real binding against the installed `ai` v6 types
```

## AC-2: The eval cascade BLOCKS fabricated numbers, wrong direction, mis-attribution, and brand violations
```gherkin
Given a generated narrative + its ReportData
When evaluateReport(narrative, reportData) runs (Stages A->D, cheap-deterministic-first)
Then Stage A re-validates the zod schema (required sections present, non-empty)
And Stage B (THE gate) builds the allowed-number set from ReportData BEFORE generation (raw metrics + deltas/percentages + formatted variants like "1,240"/"1240", "12%"/"0.12"), extracts every numeral from the prose, and BLOCKS on any unmatched numeral (F1), any sign mismatch between a matched delta and the claimed direction word (F2), or any number attributed to the wrong (source, metric, period) (F3); threshold is 100%
And Stage C brand-lints (em dash + emoji = hard BLOCK; "not just X but Y" + banned filler flagged) reusing the checkResponseSafety regex precedent in a new checkBrandStyle
And Stage D (LLM judge) is interface-present but SKIPPED in v1 (documented)
And planted-bad fixtures (hallucinated number, inverted direction, em dash, cross-source mis-attribution) each return BLOCK with the exact expected reason in unit tests
```

## AC-3: The orchestrator runs the fail-ladder and never returns an unverified narrative as passed
```gherkin
Given the writer + eval gate
When generateNarrative(reportData, deps) runs
Then a passing narrative returns { verdict:"pass", narrative, scores }
And a failing narrative triggers the ladder: regenerate up to 2 times with the specific violations quoted back to the writer, then fall back to a DETERMINISTIC template (section text with the real numbers injected, never model-generated, so always truthful and always passing Stage B), then "hold for human" if even the template path cannot be assembled
And the function NEVER returns verdict:"pass" for a narrative that did not clear the gate
```
