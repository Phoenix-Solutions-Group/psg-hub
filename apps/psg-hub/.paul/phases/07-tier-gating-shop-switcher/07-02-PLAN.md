<objective>
## Goal
Extract a single reusable, ranked tier-gate helper (`essentials < growth < performance`) from the ads-only `assertAdsTier`, and migrate the two existing ads consumers onto it with zero change in gate outcome.

## Purpose
v0.2 hard-gates customer features by Stripe tier (`essentials` / `growth` / `performance`, no migration — PROJECT constraint). Today the gate logic lives only in `assertAdsTier` and is duplicated inline in the ads page (two copies of the `SHOP_ADS_TIER_OVERRIDE` reader + two copies of the subscription read). v0.3 analytics and later customer surfaces need the same gate. This plan builds the shared mechanism so future features gate consistently instead of re-implementing per surface. It does not add new gated surfaces.

## Output
- NEW `src/lib/tier/gate.ts` — ranked tier resolver + `tierMeets` / `shopHasTier` / `assertShopTier` + centralized override reader.
- NEW `src/lib/tier/__tests__/gate.test.ts` — rank matrix, override bypass, status/missing-sub cases.
- `assertAdsTier` refactored to delegate (same `AdsApiError("tier_required")` + exact message; 5 call sites untouched).
- ads `page.tsx` refactored to use `shopHasTier(shopId, "performance")` (same `<TierGateCard/>` outcome).
</objective>

## Grounded facts (verified at plan time)
- `subscriptions` cols read today: `tier` (`essentials`|`growth`|`performance`), `status` (gate wants `"active"`), `shop_id`. RLS: `subscriptions_select = shop_id IN user_shop_ids()` (membership-clamped).
- Override env = `SHOP_ADS_TIER_OVERRIDE` (CSV of shop `slug`s). Read in TWO places today (`tier.ts` + `page.tsx`), both identical. An override-listed shop is treated as top tier.
- `assertAdsTier(shopId)` is imported at 5 ads route call sites (accounts, authorize, campaigns, campaigns/[id], campaigns/sync) and throws `AdsApiError("tier_required", "Performance tier required for Google Ads")`. Those routes map the `tier_required` code to an HTTP status — the code + message must stay identical.
- ads `page.tsx` reads the subscription with the USER client (`supabase`) today; `assertAdsTier` reads it with the SERVICE client. This plan standardizes the gate on the service client (gating must not depend on a caller's RLS visibility). Outcome is unchanged for a membership-verified shop because `subscriptions_select` already grants that exact read — and dashboard membership is verified upstream (06-03 gate + 06-05). See AC-4 and Boundaries.

## AC-1: Ranked tier mechanism is correct
```gherkin
Given the tier order essentials < growth < performance
When tierMeets(current, min) is evaluated
Then it returns true iff rank(current) >= rank(min)

```

## AC-2: Override bypass works at any minimum
```gherkin
Given a shop whose slug is in SHOP_ADS_TIER_OVERRIDE
When shopHasTier(shopId, min) or assertShopTier(shopId, min) runs for ANY min (incl. "performance")
Then the gate passes regardless of subscription tier or status (even no subscription / inactive)
And the override reader lives in ONE place (src/lib/tier/gate.ts), consumed by both ads consumers
```

## AC-3: Subscription-based gating matches today's behavior
```gherkin
Given a shop with no override
When the subscription is active AND its tier rank >= the required min
Then the gate passes
When the subscription is missing, status != "active", or tier rank < the required min
Then the gate fails (shopHasTier -> false; assertShopTier -> throws via the supplied error factory)
```

    - `export type Tier = "essentials" | "growth" | "performance"`.
    - `export const TIER_RANK: Record<Tier, number> = { essentials: 1, growth: 2, performance: 3 }`.
    - `export function tierMeets(current: Tier | null | undefined, min: Tier): boolean` — `current != null && TIER_RANK[current] >= TIER_RANK[min]`. Guard unknown strings (a DB value outside the union) -> treat as not-meeting (return false), do not throw.
    - `function overrideAllowlist(): Set<string>` — the EXACT existing reader of `process.env.SHOP_ADS_TIER_OVERRIDE` (CSV -> trimmed -> non-empty slugs). This becomes the single definition.
    - `export async function getShopTier(shopId: string): Promise<{ tier: Tier | null; active: boolean; overridden: boolean }>` — service-client read: select `id, slug` from `shops` by id (`.maybeSingle()`); compute `overridden = !!shop?.slug && overrideAllowlist().has(shop.slug)`; select `tier, status` from `subscriptions` by `shop_id` (`.maybeSingle()`); return `{ tier: sub?.tier ?? null, active: sub?.status === "active", overridden }`. If the shop row is missing, return `{ tier: null, active: false, overridden: false }` (let callers decide; do NOT throw here — the ads route's "shop not found" is the route layer's concern, see Task 2).
    - `export async function shopHasTier(shopId: string, min: Tier): Promise<boolean>` — `const s = await getShopTier(shopId); return s.overridden || (s.active && tierMeets(s.tier, min))`.
    - `export async function assertShopTier(shopId: string, min: Tier, makeError?: (message: string) => Error): Promise<void>` — if `!(await shopHasTier(shopId, min))` throw `makeError?.(\`${min} tier required\`) ?? new Error(\`${min} tier required\`)`.

Cover: TIER_RANK ordering + tierMeets matrix (incl. null current, unknown string); override slug -> shopHasTier true at min="performance" with no/inactive sub; active performance passes performance; active growth fails performance but passes growth+essentials; missing sub -> false; inactive sub -> false; missing shop -> false; assertShopTier throws the factory error on fail and resolves on pass.
