<objective>
## Goal
Add a unified active-shop context (a validated cookie) plus an MSO shop switcher in the app shell, and make the customer pages render the active shop instead of resolving one ad-hoc (`.limit(1)` / all-shops-mixed). The cookie SELECTS among a user's already-authorized shops; it never grants access ‚Äî RLS + membership stay the backstop.

## Purpose
v0.2 supports MSOs (one user, many shops) via a shop switcher (PROJECT target users). Today there is no shared shop context: settings reads an arbitrary `.limit(1)` shop, reviews and content render every member shop mixed together, ads bounces through `?shop_id=`. An MSO has no way to focus the dashboard on one shop. This plan adds the switch mechanism + shell UI and points the clean-win customer surfaces at it. It closes Phase 7.

## Output
- NEW `src/lib/shop/context.ts` ‚Äî user-shops resolver, the `psg_active_shop` cookie name, a PURE `resolveActiveShop` (re-validates the cookie against current membership), and `getActiveShopContext(userId)`.
- NEW `POST /api/shop/switch` ‚Äî membership-validated cookie setter.
- NEW `src/components/dashboard/shop-switcher.tsx` ‚Äî client switcher (0 ‚Üí hidden, 1 ‚Üí static label, 2+ ‚Üí dropdown).
- Switcher wired into `dashboard/layout.tsx`; settings / reviews / content filtered to the active shop.
</objective>

<acceptance_criteria>

## AC-1: Active-shop resolution re-validates the cookie against current membership
```gherkin
Given a user's current shop memberships and a cookie value
When resolveActiveShop(shops, cookieValue) runs
Then it returns the cookie's shop id ONLY if that id is in the current membership set
And otherwise falls back to the owner-role shop, else the first shop, else null
And a stale cookie (shop the user is no longer a member of) never selects that shop
```

## AC-2: The switch route validates membership before setting the cookie
```gherkin
Given an authenticated user
When POST /api/shop/switch is called with { shop_id }
Then it returns 401 if unauthenticated, 400 if shop_id is missing,
     403 if shop_id is not one of the caller's memberships,
     and 200 + a Set-Cookie (psg_active_shop, httpOnly, Secure, SameSite=Lax, path=/) on success
And the cookie is never trusted for authorization ‚Äî only membership decides 200 vs 403
```

## AC-3: The switcher renders the right control for the membership count
```gherkin
Given the app shell
When a user has 0 shops (e.g. staff) Then no switcher renders
When a user has exactly 1 shop Then a static shop-name label renders (no dropdown)
When a user has 2+ shops Then a dropdown renders; selecting a shop POSTs /api/shop/switch then refreshes
And the client component receives shops + activeShopId as props and never reads the cookie itself
```

## AC-4: Customer pages render the active shop
```gherkin
Given a user with an active shop
When they open Settings, Reviews, or Content
Then Settings shows that shop's profile (not an arbitrary .limit(1) shop)
And Reviews and Content are filtered to .eq("shop_id", activeShopId) (not all member shops mixed)
And switching shops (via the switcher) changes what these pages show after refresh
```
