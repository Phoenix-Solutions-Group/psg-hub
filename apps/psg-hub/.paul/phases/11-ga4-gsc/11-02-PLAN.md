<objective>
## Goal
Build the GA4 daily-ingest vertical: read a shop's linked `ga4` `google_oauth_accounts` row (from 11-01),
decrypt its shared refresh token, build a `BetaAnalyticsDataClient`, and pull account-level daily traffic
metrics via ONE trailing-window `runReport` per property — `sessions`, `totalUsers`, `activeUsers`,
`keyEvents` (the 2024 conversions rename), `engagementRate` — mapping each returned day into one
source-agnostic `analytics_snapshots` row (`source='ga4'`, `period='daily'`). A CRON_SECRET-gated cron
drives it idempotently; an additive GA4 panel renders it on `/dashboard/analytics`. ZERO prod contact,
NO migration (the snapshot/ledger CHECK already admits `ga4`).

## Purpose
11-01 shipped the OAuth foundation (one consent, one refresh token, linked GA4 + GSC accounts). This plan
turns the linked GA4 property into stored, surfaced traffic data — the second source in the unified v0.3
analytics view and a required input to the Phase-12 PSG report. It mirrors the shipped Phase-10 Google Ads
vertical structurally (link -> decrypt -> trailing-window fetch -> idempotent snapshot -> additive panel),
so the only genuinely new surface is the GA4 Data API contract itself.

## Output
- EXTEND `google-oauth/accounts.ts`: a shared `getLinkedAccount(shopId, source)` (row read + `\x`hex bytea
  decode + decrypt, mirroring `getGoogleAdsClient`) and a generic `markAccountError(accountId, message)`.
- NEW `google-oauth/ga4-client.ts` (`BetaAnalyticsDataClient` via the isolated `authClient` injection),
  `ga4-metrics.ts` (`fetchGa4DailyMetrics` -> `Map<dateISO, Ga4Metrics>`), `ga4-sync.ts` (`syncGa4Snapshots`).
- NEW `/api/cron/ga4-sync` route (+ `vercel.json` entry, staggered `30 6`) and a `Ga4Metrics` type.
- An additive GA4 organic-traffic panel on `/dashboard/analytics` (own unlinked state; organic + paid blocks
  byte-untouched).
- Unit tests (metrics contract, orchestrator containment, cron gate, account decode) + an e2e ga4 round-trip.
- At UNIFY: 11-02-SUMMARY. LIVE activation (real GA4 runReport, the `authClient` runtime smoke, GA4 Admin/Data
  API enablement, consent publish, deploy) is the shared Phase-11 operator gate batch, recorded as a deferral.

## AC-1: GA4 Data client from a linked account (shared read + decrypt + authClient injection)
```gherkin
Given a shop has a status='linked' google_oauth_accounts row with source='ga4' (external_account_id
      'properties/<numeric>', a `\x`hex-encoded shared refresh token, key_version)
When getLinkedAccount(shopId, 'ga4') runs
Then it reads the row with a service client, decodes the bytea `\x<hex>` text form to a Buffer (Buffer +
     ArrayBuffer fallbacks, mirroring getGoogleAdsClient l.181-187), decryptRefreshToken's it with key_version,
     and returns { accountId, externalAccountId, refreshToken } (or null when no linked row)
And the eligibility read is DETERMINISTIC — exactly ONE ga4 row per shop (order by linked_at desc, pick first)
     so multi-property-per-shop can never silently double-write the (shop,source,date,period) key
And getGa4DataClient(shopId) builds a BetaAnalyticsDataClient via buildOAuth2Client + the gax `authClient`
     field (the 11-01 ga4-enumerate idiom — NOT googleapis `auth`), returning { client, property, accountId }
And markAccountError(accountId, message) flips google_oauth_accounts.status='error' + last_error (sanitized),
     mirroring markAccountAuthFailed but on the generic table (reused by GSC in 11-03).
```

## AC-2: fetchGa4DailyMetrics — one trailing-window runReport, date-map of daily totals
```gherkin
Given a built GA4 Data client for property 'properties/<id>' and a {startDate, endDate} window (YYYY-MM-DD)
When fetchGa4DailyMetrics(shopId, {startDate, endDate}, deps?) runs
Then it issues ONE runReport with dimensions:[{name:'date'}], NO channel dimension (account-level daily totals,
     one row per day — RESEARCH quota: prefer one trailing-window call over N single-day calls), metrics
     sessions/totalUsers/activeUsers/newUsers/engagedSessions/keyEvents/engagementRate, orderBys date asc,
     keepEmptyRows:false, returnPropertyQuota:true
And it uses `keyEvents` as the conversions metric (NOT the deprecated `conversions`)
And every returned metricValue STRING is Number()'d before storing, and the `date` dimension's YYYYMMDD value is
     reformatted to YYYY-MM-DD before it becomes the snapshot key
And it returns Map<dateISO, Ga4Metrics> (a 0-row window yields an empty Map — the empty-day path)
And response metadata samplingMetadatas / subjectToThresholding / dataLossFromOtherRow are logged per ingest
     (console.warn — there is NO per-call ledger; ads_api_call_log is NOT reused), and the propertyQuota is logged
And every upstream call is wrapped in CircuitBreaker + withRetry; failures map through mapGoogleApiError to a typed
     GoogleApiError (auth_failed propagates, transient retries) — unit tests cover map/string-coercion/date-reformat/
     keyEvents/empty-window/error-path.
```

## AC-3: syncGa4Snapshots orchestrator + CRON_SECRET cron (idempotent, contained, source='ga4')
```gherkin
Given linked ga4 accounts across N shops and a trailing window GA4_RESYNC_DAYS (default 3 — NOT the ads 7;
      GA4 reprocesses ~18-48h, RESEARCH freshness)
When syncGa4Snapshots(service, options) runs
Then eligibility is ONLY status='linked' source='ga4' accounts, ONE per shop (deterministic, AC-1); each shop gets
     ONE fetchGa4DailyMetrics over [yesterday-(GA4_RESYNC_DAYS-1) .. yesterday], fanned into one
     AnalyticsSnapshotInsert per returned date (source='ga4', period='daily'), upserted via upsertSnapshots
     (onConflict shop_id,source,date,period — re-running the window nets zero new rows)
And a single shop's failure is CONTAINED (failed++, batch continues); an auth_failed calls markAccountError so the
     surface shows "needs re-link"; one analytics_sync_runs ledger row (source='ga4') opens/closes per run
And GET+POST /api/cron/ga4-sync gate on `Authorization: Bearer ${CRON_SECRET}` via timingSafeEqual BEFORE any client
     build (401 incl. unset secret), return 503 when the GA4 OAuth creds are absent (GOOGLE_OAUTH_CLIENT_ID/SECRET +
     GOOGLE_ANALYTICS_OAUTH_REDIRECT_URI — NO developer token, unlike ads), 200 with {synced,skipped,failed} otherwise
And the route declares `export const runtime = "nodejs"` (the gax/grpc GA4 client is not Edge-safe — the ads cron
     template OMITS this; it must be ADDED here)
And vercel.json adds {path:'/api/cron/ga4-sync', schedule:'30 6 * * *'} (staggered after semrush 0 6 + ads 15 6).
```
