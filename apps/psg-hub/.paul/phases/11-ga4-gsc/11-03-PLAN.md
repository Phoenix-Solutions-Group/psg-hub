<objective>
## Goal
Build the GSC daily-ingest vertical — the THIRD and final source of Phase 11. Read a shop's linked `gsc`
`google_oauth_accounts` row (from 11-01), decrypt its shared refresh token (the SAME token GA4 uses — one
combined consent), build a `google.searchconsole` client, and pull site-level daily search metrics via ONE
trailing-window `searchanalytics.query` per site — `clicks`, `impressions`, `ctr`, `position` — mapping each
returned day into one source-agnostic `analytics_snapshots` row (`source='gsc'`, `period='daily'`). A
CRON_SECRET-gated cron drives it idempotently; an additive "Search performance" panel renders it on
`/dashboard/analytics`. ZERO prod contact, NO migration (the snapshot/ledger CHECK already admits `gsc`).

## Purpose
11-01 shipped the OAuth foundation (one consent, one refresh token, linked GA4 + GSC accounts) and 11-02
shipped the GA4 ingest. This plan adds the last source, completing the Phase-11 build: a linked GSC site's
organic search performance becomes stored, surfaced data — the third input to the unified v0.3 analytics view
and a required input to the Phase-12 PSG report. It mirrors the GA4 vertical structurally (link -> decrypt ->
trailing-window fetch -> idempotent snapshot -> additive panel); the genuinely new surface is the GSC Search
Analytics API contract, which has a DIFFERENT response shape than GA4's `runReport` (see AC-2 — do NOT clone
the ga4-metrics parser).

## AC-1: GSC client from a linked account (shared read + decrypt + googleapis `auth` — NOT gax)
```gherkin
Given a shop has a status='linked' google_oauth_accounts row with source='gsc' (external_account_id a site key
      'sc-domain:<host>' OR 'https://.../', the SAME `\x`hex-encoded shared refresh token GA4 uses, key_version)
When getGscClient(shopId, deps?) runs
Then it calls the SHARED getLinkedAccount(shopId, 'gsc') (already built in 11-02 — read + bytea `\x`hex decode +
     decrypt + deterministic ONE row per shop); if null it throws GoogleApiError('auth_failed', 'No linked GSC site')
And it builds the client by MIRRORING gsc-enumerate.ts — googleapis' OWN `new google.auth.OAuth2(clientId,
     clientSecret, redirectUri)` + `setCredentials({refresh_token})`, then `google.searchconsole({version:'v1',
     auth})` — passing the OAuth2Client via the `auth` field (NOT the gax `authClient`; do NOT reuse
     buildOAuth2Client / the ga4-client BetaAnalyticsDataClient construction)
And it returns { client, siteUrl: externalAccountId, accountId }, with an injectable deps seam for tests.
```

## AC-2: fetchGscDailyMetrics — one trailing-window searchanalytics.query, date-map of daily totals (OWN parser)
```gherkin
Given a built GSC client for site '<siteKey>' and a {startDate, endDate} window (YYYY-MM-DD, interpreted PT)
When fetchGscDailyMetrics(shopId, {startDate, endDate}, deps?) runs
Then it issues ONE searchanalytics.query with siteUrl as a METHOD param (NOT a body field) and a body of
     { startDate, endDate, dimensions:['date'], type:'web', dataState:'final', rowLimit:25000 } — siteUrl defensively
     wrapped in encodeURIComponent (RESEARCH UNVERIFIED #4: googleapis auto-encoding is version-dependent), the live
     probe deferred
And it parses the GSC response shape — rows are `{ keys: ['YYYY-MM-DD'], clicks, impressions, ctr, position }`:
     there are NO metricHeaders / metricValues, the date in keys[0] is ALREADY YYYY-MM-DD (do NOT apply any
     YYYYMMDD->ISO reformat — that would corrupt it), and clicks/impressions/ctr/position arrive as JSON NUMBERS
     (coerce defensively with Number(), but the active GA4 string-coercion+reformat steps do NOT carry over)
And it returns Map<dateISO, GscMetrics> keyed by keys[0] (a 0-row window — common for recent lagging days — yields
     an empty Map, the empty-day path)
And every upstream call is wrapped in CircuitBreaker + withRetry (reuse the ga4-metrics isRetryable: code in
     timeout/upstream/rate_limited); failures map through mapGoogleApiError (already classifies Gaxios HTTP status
     from 11-01) to a typed GoogleApiError (auth_failed propagates, transient retries)
And unit tests cover: the request body (date dims, type=web, dataState=final), keys[0]-as-date parse, numeric
     fields, empty-window, and the error path — via the injectable query seam (no live call).
```
