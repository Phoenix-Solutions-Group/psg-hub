<objective>
## Goal
Build the shared Google OAuth foundation for Phase 11: a single combined-scope consent
(`analytics.readonly` + `webmasters.readonly`) that links a shop's Google account, enumerates the
user's GA4 properties (Admin `accountSummaries.list`) AND Search Console sites (`sites.list`), and
lets the operator pick one of each via the proven 10-04 two-step picker. The chosen GA4 property and
GSC site are persisted as two rows in a NEW generic `google_oauth_accounts` table, sharing ONE
encrypted refresh token. No ingest, no panels yet — those are 11-02 (GA4) and 11-03 (GSC), which both
depend on this account model.

## Purpose
Phases 11-02/11-03 (the GA4 + GSC daily ingests) cannot run without a stored, per-shop refresh token
and a selected property/site. The research confirms one consent yields one refresh token usable for
both APIs, so a single shared link + a generic account table is the smallest correct foundation. This
extends the v0.3 analytics surface toward the unified SEMrush + Google Ads + GA4 + GSC view.

## Output
- 2 NEW migrations (LOCAL-applied only): `google_oauth_accounts` + `google_oauth_pending_states`.
- A NEW `src/lib/google-oauth/` module: a parameterized clone of the Ads OAuth state machine (scope +
  redirect generalized), a shared `OAuth2Client` builder, GA4 + GSC enumeration, and the persist helper.
- The 3 link routes (`authorize` / `callback` / `select`) + a link-entry button on the analytics surface.
- Unit tests for enumeration + state carry; an e2e schema round-trip proving the bytea token + RLS.
- At UNIFY: 11-01-SUMMARY. Live activation (real Google link, deploy) is the Phase-11 gate batch, NOT this plan.
</objective>
