<objective>
## Goal
Reconcile the psg-hub **reviews surface** (read + AI-response governance) from its inherited-but-undeployed BSM schema assumptions onto the live deployed schema, so the surface renders and operates without runtime errors and is gated by the 06-02/06-03 RBAC spine.

## Purpose
Per `06-RESEARCH-DOSSIER.md` (W3), every inherited customer surface errors at runtime today because the code reads tables/columns that do not exist on the live shared DB. Reviews is one of two such surfaces (ads = 06-05). This plan makes reviews real: it is the first customer feature to actually work end-to-end on deployed schema, and it exercises the RBAC spine (shop membership + role gating) the rest of v0.2 depends on.

## Output
- Reviews page + 3 reviews API routes + responder/types/components repointed to live `review_items` / `review_responses` / `shop_users`.
- One operator decision (schema fork) resolved and, if "extend", a single idempotent migration applied to prod under the S1 protocol.
- Empty-state render proven (both review tables are 0 rows today).
- `06-04-SUMMARY.md`.
</objective>

  <decision>How to reconcile `review_responses`: the app implements a full AI-response governance model (versioning + optimistic concurrency, safety flags/override + attribution, tone preset, model/prompt provenance, approval attribution) that the live bare table lacks.</decision>
  <context>
    Live `review_responses` = {id, review_item_id, draft_text, status, published_at, created_at, updated_at}.

    review_responses + review_items are BOTH 0 rows → any schema change is data-risk-free. This decision shapes Tasks 2-4.
  </context>
  <options>
    <option id="extend">
      <name>Path A — Extend live schema to the app's governance model (RECOMMENDED)</name>
      <pros>Preserves shipped + tested BSM governance (version concurrency, safety override audit, approval attribution). Code change is mechanical column-name reconcile only. 0 rows = zero migration risk. Consistent with 06-02 (extend-under-S1 precedent).</pros>
      <cons>Schema change on shared prod (314k PII rows in OTHER tables, none in these). Must pass S1 protocol + advisor diff. Adds ~10 columns to one table.</cons>
    </option>
    <option id="strip">
      <name>Path B — Strip the app to the live bare table</name>
      <pros>No prod schema change.</pros>
      <cons>Throws away working governance features (no version concurrency, no safety-override audit trail, no approval attribution, no tone persistence). Large feature regression of shipped work; rewrites response-modal + responder + both routes deeply. Not recommended.</cons>
    </option>
    <option id="hybrid">
      <name>Path C — Extend only the governance-critical columns, derive/drop the rest</name>
      <pros>Smaller migration (keep version + safety_flags + approved_at + shop_id; drop model_id/prompt_version provenance to app-log or omit).</pros>
      <cons>Mixed code paths (some fields persisted, some not); more reconcile reasoning than A for marginal column savings. Pick only if Nick wants provenance columns out of the DB.</cons>
    </option>
  </options>
