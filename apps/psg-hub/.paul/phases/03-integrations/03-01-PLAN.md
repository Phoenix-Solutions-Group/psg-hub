<acceptance_criteria>

## AC-1: Resilient mail send
```gherkin
Given a configured SendGrid adapter (SENDGRID_API_KEY set) and a valid email payload
When sendEmail() is called and the SendGrid API returns a transient error (429 or 5xx)
Then the call is retried with exponential backoff (bounded), and a permanent error (4xx other than 429) is NOT retried but surfaced as a typed MailError
And after the circuit breaker's failure threshold is exceeded, further calls short-circuit (fail fast) without hitting the network
And no failure path uses a bare catch — errors are typed, logged, and either retried or rethrown
```

## AC-2: Idempotent, verified event webhook
```gherkin
Given the SendGrid Event Webhook posts a batch of events with a valid ECDSA signature
When POST /api/webhooks/sendgrid receives it
Then the signature is verified against SENDGRID_WEBHOOK_VERIFICATION_KEY (invalid/missing → 400, like the Stripe route)
And each event is persisted to email_events keyed by sg_event_id, where a duplicate sg_event_id is ignored (idempotent replay → no duplicate row)
And the endpoint returns 200 only after successful persistence (so SendGrid retries on a true failure, which the unique key makes safe)
```

## AC-3: Domain authentication + live delivery (operator-gated)
```gherkin
Given the operator has created a SendGrid API key, completed Domain Authentication for psgweb.me (SPF/DKIM CNAMEs added + verified), configured a branded sender, and registered the Event Webhook with signature verification enabled
When a test email is sent via sendEmail() to a real inbox
Then the email is received from a psgweb.me sender, and a corresponding "delivered" event row appears in email_events
```
