# ADR-001: Re-platform FileMaker behavior, do not port code or schema 1:1

## Status

Proposed

## Context

PSG runs the FileMaker Advantage Program (four files: Advantage, Survey, Import
Flush, Web) as its operational daily driver. The DDR analysis of those files
exposed heavy structural duplication: 325 per-shop import scripts, triplicate
per-letter scripts (Preview/Print/Loop), and three live generations of survey
alert scripts (v2/WIP/v3 LIVE). It also confirmed there is almost no event
layer to preserve (0 layout triggers, 12 `OnObjectExit` validations, 1 in-file
PSOS). Meanwhile `PLANNING.md` already commits to replacing FileMaker and already
drafts the target tables (`repair_customers`, `survey_responses`,
`production_batches`, the 26 reports) and the cron surfaces. The one gap is the
behavioral spec for what those modules must do.

A direct FileMaker-to-SQL lift-and-shift would reproduce that duplication and
import a schema shaped by FileMaker constraints rather than by the psg-hub
domain model.

## Decision

Re-platform behavior, do not port code or schema 1:1. Rebuild each FileMaker
subsystem as a psg-hub module (Ops, Surveys, Production, Reports) on the
already-planned Supabase schema. Use the DDR analysis as the behavioral source
of truth for the rules that must be preserved exactly: letter eligibility
windows, the Perfect/Misfire/HotSpot/Unresolved/Referral alert classifications
and thresholds, the suppression rule, and the printed/posted idempotency
semantics. Collapse the duplicated FileMaker constructs into single,
mapping-driven or status-driven implementations.

## Consequences

**Positive**

- One importer (`import_templates`) replaces 325 per-shop scripts.
- One alert classifier replaces three live generations of alert scripts.
- Batch + template + status replaces the triplicate per-letter scripts.
- The schema is shaped by the psg-hub domain model, not by FileMaker layout
  constraints, and integrates cleanly with the shared Supabase project.

**Negative / trade-offs**

- More upfront design work than a mechanical lift-and-shift.
- Correctness depends on the DDR being an accurate behavioral record; the v3 LIVE
  alert logic must be confirmed authoritative before it is encoded.
- A behavioral parity check is required at cutover to prove the rebuild matches
  FileMaker output.

## Alternatives considered

- **Lift-and-shift FileMaker schema and scripts into Postgres.** Rejected: it
  reproduces the duplication the DDR flagged as technical debt and locks psg-hub
  into a FileMaker-shaped data model.
- **Keep FileMaker and bolt psg-hub on as a read-only mirror.** Rejected: it does
  not retire the legacy system, which is the explicit goal of the milestone plan.
