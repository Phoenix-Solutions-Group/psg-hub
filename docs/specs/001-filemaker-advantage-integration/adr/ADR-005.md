# ADR-005: Historical data migration is optional, via a psg-data-lake worker (v1.3.5)

## Status

Proposed

## Context

psg-hub replaces FileMaker Advantage as the daily driver, but the functional
replacement (v1.1 ops foundation through v1.4 reports) does not inherently require
the historical data to move with it. FileMaker holds large history (the DDR counts
roughly 281K repair customers and 334K survey responses). Moving that data is a
distinct effort with its own risks: match-key resolution to real foreign keys
(ADR-2), PII redaction, dedupe, and count reconciliation. `PLANNING.md` already
treats this as add-on scope (Q2/Q15 deferred), and the SDD keeps functional
replacement off the critical path of data movement.

There is an existing Python ETL home (`psg-data-lake`) and existing extract
sources (the DDR CSV exports plus the FileMaker Data API on the psgweb.me host).

## Decision

Treat historical data migration as optional v1.3.5 scope, delivered by a
`psg-data-lake` Python worker (`fm_migrate.py`: extract → transform → load). It
sources from the DDR CSV exports and the FileMaker Data API, resolves legacy match
keys to real foreign keys, applies PII redaction, dedupes, and reconciles loaded
counts against the DDR before each sequencing step proceeds. Functional
replacement (v1.1 through v1.4) ships independently of this worker; v1.3.5 runs only if
the business requires historical continuity in psg-hub.

## Consequences

**Positive**

- Cutover is not blocked by data movement; FileMaker can go read-only for history
  while psg-hub owns current operations.
- The migration reuses the existing `psg-data-lake` runtime and the DDR exports,
  rather than standing up new infrastructure.
- Loaded counts reconcile to the DDR (repair_customers ~281K, survey_responses
  ~334K) with a 50-link spot check, giving a measurable integrity gate.

**Negative / trade-offs**

- A period exists where current data lives in psg-hub and deep history lives in
  read-only FileMaker, so cross-period reports must account for two sources until
  the migration completes.
- If the business later requires history, the migration becomes a real project
  with its own timeline; deferring it now defers that cost, it does not remove it.

## Alternatives considered

- **Migrate all history before cutover (block cutover on data movement).**
  Rejected: it couples functional replacement to a large, risky ETL and delays
  retiring FileMaker as the daily driver.
- **Never migrate; leave all history in read-only FileMaker permanently.**
  Rejected as the default: it may be acceptable for some history, but hard-coding
  it forecloses historical continuity the business may need. Making migration an
  optional, ready-to-run phase keeps the choice open.
