# ADR-003: Move automation to Vercel cron + idempotent Postgres workers

## Status

Proposed

## Context

All automation in FileMaker Advantage today runs from the Web file: it fires
Perform Script on Server (PSOS) on a FileMaker Server schedule, and it tracks
incremental work with date-stamp fields (process a record, stamp it, skip
already-stamped records on the next run). The DDR confirmed there is no
event-trigger layer to preserve (0 layout triggers, 1 in-file PSOS). This makes
automation a single point of failure: if the scheduled PSOS does not run, nothing
self-heals, and the date-stamp pattern depends on every write succeeding exactly
once.

psg-hub targets Vercel hosting with Vercel Cron and a shared Supabase Postgres,
and `PLANNING.md` already declares the cron surface (`survey-ingest`,
`production-batch-status`, `mail-vendor-poll`, and others).

## Decision

Move all automation to Vercel cron plus idempotent Postgres workers. Retire the
Web PSOS scheduler and the date-stamp incremental pattern. Each scheduled job is
window-based and idempotent: it selects rows in a bounded date window, resolves
foreign keys, computes or classifies, and upserts under a unique constraint
(`letter_eligibility` unique on customer + letter_kind + period_key; status
columns plus `printed_at` guards for production). Idempotency replaces date stamps;
a unique constraint plus `on conflict do nothing`/`do update` prevents
double-effects.

## Consequences

**Positive**

- A missed cron self-heals on the next run because each job is window-based, not
  pointer-based.
- Re-triggering a job cannot double-send: unique constraints and `printed_at`
  guards make every run idempotent.
- Cron runs emit counts (computed, eligible, suppressed, printed) for
  observability, which FileMaker lacked.

**Negative / trade-offs**

- A missed cron only self-heals if the window is wide enough to recover the gap;
  windows must be sized deliberately.
- Loss of self-healing must still be detected: a freshness alert is required when
  `computed_at` is stale beyond about 36 hours.
- Service-role cron writes bypass RLS, so the worker code carries the
  authorization responsibility the database would otherwise enforce.

## Alternatives considered

- **Database triggers / pg_cron inside Postgres.** Rejected: it spreads
  automation logic across the database and the app, and is harder to observe and
  deploy than route handlers the team already owns.
- **Port the FileMaker date-stamp pattern to Postgres columns.** Rejected: it
  keeps the brittle exactly-once-write assumption and does not give the
  self-healing that window-based idempotent runs provide.
