# ADR-002: Replace the composite string match key with a real foreign key

## Status

Proposed

## Context

In FileMaker, a survey response is linked to its repair customer through a
composite string match key built from shop id + month + year
(`RC_MatchField_Master`). The DDR cross-file analysis showed this link breaks
silently: any mismatch in shop id, month, or year produces no error, just an
unmatched survey that disappears from the feedback loop. Because the survey↔letter
loop drives customer contact and revenue (suppression and referral letters depend
on it), a silent break is a correctness and revenue risk, not just a data-quality
nuisance.

psg-hub stores survey responses and repair customers as first-class rows in one
Supabase schema, so a referential link is available.

## Decision

Replace the composite string match key with a real foreign key:
`survey_responses.repair_customer_id` references `repair_customers(id)`. The FK is
nullable until matched. A survey that cannot be resolved is retained with a null
FK and surfaced in a needs-match ops queue; it is never silently dropped. The
legacy `RC_MatchField_Master` string is kept as `repair_customers.legacy_match_key`
(indexed) for migration reconciliation only, and as a fallback resolver during the
v1.3.5 historical load.

## Consequences

**Positive**

- Broken links become visible and queryable instead of silent.
- The needs-match queue gives ops a place to resolve unmatched surveys, with a
  target unmatched rate under 2% after migration reconciliation.
- Referential integrity is enforced by the database, not by string assembly in
  application logic.

**Negative / trade-offs**

- The v1.3.5 migration must resolve every legacy composite key to a real FK,
  using `legacy_match_key` as the bridge.
- A nullable FK plus a needs-match workflow is one more state to operate and
  monitor.

## Alternatives considered

- **Keep the composite string key in Postgres.** Rejected: it carries the silent
  break forward and gives the database no way to enforce the relationship.
- **Hard FK with NOT NULL, reject unmatched surveys at write time.** Rejected: it
  would drop or block legitimate surveys that arrive before their repair customer
  is matched, recreating the very data loss this decision removes. Nullable FK
  plus a needs-match queue preserves every survey.
