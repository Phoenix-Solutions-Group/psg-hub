# v1.3 — FileMaker Advantage daily-driver cutover runbook

**Issue:** PSG-44 · **Milestone:** v1.3 Production Module (Lob path) · **Owner:** Ada (Chief Developer)
**Decision basis:** PLANNING.md D32 / Q2 — *FileMaker Advantage retired as daily driver at v1.3; psg-hub
Production module replaces it for all new production. FileMaker stays read-only for historical access.
Full data migration is the optional v1.3.5 add-on (Q15) and is **out of scope here.***

## 1. Scope

**In scope (this cutover):**

- Point the PSG production team to **psg-hub** for **all new production** (thank-you / warranty / envelopes).
- Leave **FileMaker Advantage running read-only** for historical lookups — no writes, no new batches.
- Verify the end-to-end production happy path before flipping.

**Out of scope:**

- Full historical data migration from FileMaker → psg-hub. That is the optional **v1.3.5** add-on (Q15).
  Until/unless it ships, history lives in FileMaker (read-only) and pre-cutover batches are *not* searchable
  inside psg-hub.

## 2. The end-to-end happy path (the thing that must work before we flip)

> company has program → repair customer assigned → batch generated → printed → moved to historical → reprintable

| Step | Surface | Backing |
|------|---------|---------|
| Company + program | `/ops/companies`, `/ops/sys-config/products` | `companies`, `products` |
| Repair customer | `/ops/repair-customers` | `repair_customers` (FK → company) |
| Batch generate | `/ops/production` → New batch | `POST /api/production/batches` → `production_batches` (`draft`) |
| Print (single or batch) | `/ops/production` print action | `POST /api/production/documents/[id]/print` → Lob via `MailAdapter`; status → `printing`/`historical` |
| Historical | `/ops/production` historical search | `GET /api/production/documents` (by print id / company / product / repair customer / status) |
| Reprint (audited) | historical → reprint | `POST /api/production/documents/[id]/reprint` → `production_reprint_log` |

Vendor selection is vendor-agnostic through `MailAdapter` (`src/lib/production/types.ts`); Lob is the live
vendor, in-house print queue is the fallback. **Live Lob spend is gated by board gate G4** — a `test_*`
`LOB_API_KEY` exercises the whole path with zero per-piece spend; the `live_*` key is only set in prod env
after G4 is approved.

## 4. Cutover procedure (flip day)

1. Confirm every go-condition in §3 is checked.
2. Operator sets the prod `LOB_API_KEY` to the `live_*` key and confirms `LOB_WEBHOOK_SECRET` is set
   (Lob dashboard → Webhooks). Deploy/restart so the new env is live.
3. Smoke test in prod with a **single** low-cost piece: generate a 1-document batch → print → confirm Lob
   accepts (`external_id` returned, status advances) → confirm the Lob webhook updates status →
   reprint once → confirm it appears in historical search. (One real piece = bounded live spend.)
4. Announce cutover to the production team. FileMaker switched to **read-only** (no new batches there).
5. Monitor the Lob webhook + production queue for the first full production day.

## 5. FileMaker read-only posture

- FileMaker Advantage stays available for **historical lookups only**.
- No new batches, edits, or production runs in FileMaker after the flip.
- Communicate the read-only date so the team knows the clean line between "history = FileMaker" and
  "new production = psg-hub".
- Historical *continuity inside psg-hub* (pre-cutover batches searchable in-hub) requires the v1.3.5 data
  migration — explicitly deferred.

## 6. Rollback

The flip is **org/process + one env var**, not a destructive data migration, so rollback is cheap:

1. Operator reverts prod `LOB_API_KEY` to the `test_*` key (halts live spend immediately).
2. Tell the production team to resume new production in FileMaker (it is still intact — we only made it
   read-only by convention, not by deletion).
3. No data unwind needed: psg-hub batches created during the window remain in psg-hub; nothing was removed
   from FileMaker.

Trigger rollback if: Lob rejects a material share of pieces, webhook status updates stop, or the team hits a
blocking workflow gap in the first production day.

## 7. Go / No-Go

**GO** when all §3 go-conditions are checked — most importantly: production E2E green in CI/staging,
PII sign-off, **and G4 approved**.

**NO-GO / hold in test mode** if G4 is not yet approved, the production E2E is not green, or PII review has
not signed off. In NO-GO state psg-hub still runs end-to-end in **Lob test mode** (no live spend), so the
team can train and we can validate without flipping FileMaker.
